📦 The booking app

The app you'll deploy to AWS three different ways. Zero dependencies — Node's standard library only. Files: server.js · tours.json · user-data.sh · public/index.html

Try it locally first (2 minutes, before any AWS)

node server.js
# → http://localhost:3000

Deploy shapes by milestone

MilestoneFrontendAPIHow
M1 rehostEC2 (this app serves both)same EC2user-data.sh at launch
M2 replatformS3 static website (public/ + config.js)EC2 / ALBwindow.API_BASE = "http://YOUR-ALB-DNS"
M4 refactorS3 static websiteAPI Gateway + Lambda + DynamoDBeach handleApi branch becomes a Lambda handler

Endpoints

MethodPathPurpose
GET/api/healthstatus + which instance served you (rotates behind the ALB)
GET/api/configcompany branding
GET/api/tourstour catalog
GET/api/tours/:idone tour
POST/api/bookingsbook: {"tourId":1,"name":"Ada","guests":2}
GET/api/bookingslist bookings (in-memory — M4 asks why that's a problem)
GET/api/statsrequest metering — evidence for your cost/usage reports

API responses send permissive CORS headers so the S3-hosted frontend can call them (journal question: why is * fine here, risky in production?). Booking state is in-memory on purpose — it vanishes on every instance replacement, which is exactly the lesson that motivates DynamoDB in M4.
